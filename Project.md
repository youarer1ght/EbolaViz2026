# 数据可视化导论 — 课程 Project 要求

> 课程大作业（占总评 **50%**）

本次课程大作业为设计开发可视分析系统（可组队，2~3 人），自由选题。具体的形式可以参考 [ChinaVis 2025 挑战赛]、[ChinaVis 2024 挑战赛]、[VAST Challenge 2025] 等，但需要使用不同的数据。

---

## 一、组队与分工

- 每组 2~3 人，不接受 4 人及以上组队。
- 提交材料中需包含**分工说明**，列出每位成员负责的模块（数据处理、前端开发、可视化设计、系统集成、文档撰写等）。
- 答辩时每位组员必须主讲一部分，并接受针对该部分的提问。
- 代码仓库需保留 git 提交记录（或类似的过程证据），作为贡献度判断的辅助依据。

## 二、数据要求

- 不得使用与 ChinaVis / VAST 等参考挑战赛完全相同的数据；可选用公开数据集、自采数据或合作方数据，需在文档中说明来源与许可。
- 数据规模与复杂度需足以支撑多视图分析，建议不少于 **1 万条记录**，或具备时空属性、层级结构、多实体关系等特征。
- 禁止使用 iris、titanic、tips 等教学玩具数据集。

## 三、LLM 使用政策

本次作业允许且鼓励使用大语言模型辅助编码、调试、查阅文档、生成初版设计草案等。但以下要求是硬性的：

1. 系统的核心设计决策（包括分析任务的界定、视图选择与协调方式、关键交互的设计理由）必须由团队自行做出，并能在答辩中清晰解释。
2. 系统文档中需包含一节 **AI 使用声明**，列出使用了哪些工具、分别承担什么角色，并简要反思哪些决策由团队做出、哪些来自模型建议。
3. 答辩问答环节会针对具体设计决策提问，若团队无法解释自己系统中的关键选择，将直接影响展示分与工程分。

## 四、创意与参考边界

- 允许并鼓励借鉴已发表论文或开源项目的设计思想与组件（如 LineUp、Parallel Sets、TimeNotes 等经典设计的形态与变体），但必须在系统文档中注明来源。
- 系统必须包含团队自己的设计贡献：针对新数据的可视化设计适配、新的分析任务、新的视图组合、新的交互形式等。**对现有开源项目的简单换数据不视为有效贡献。**

## 五、交付物与可复现性要求

- 代码仓库需包含 `README.md`，说明环境依赖、启动命令、数据加载方式。
- 助教应能在 **15 分钟内**完成环境搭建并运行起系统；若系统依赖复杂，建议部署一个在线可访问版本（例如 GitHub Pages 等）。
- 鼓励在系统文档中提供 1~2 个 case study，展示如何通过系统发现数据 pattern。

## 六、评分标准（共 50%）

### 1. 中期文档（5%）

包含项目背景、分析任务定义、数据说明、设计方案、目前实现进展、待完成的工作。

### 2. 系统文档（5%）

包括设计需求、设计介绍、案例展示（case study）、AI 使用声明。

### 3. 工程代码（30%）

| 评分项 | 分值 | 说明 |
|--------|:----:|------|
| 分析任务的明确性与价值 | 5% | 系统解决了什么数据分析问题？为什么值得用可视化而非纯统计/机器学习方法解决？ |
| 可视化设计质量 | 8% | 图表类型与分析任务的匹配、视觉通道运用、针对数据特性的设计创新 |
| 多视图协调与交互设计 | 8% | 视图间的语义关联、brushing & linking、details-on-demand 等是否到位，视图之间是否真正联动而非各自独立 |
| 数据处理与系统完整度 | 5% | 数据规模与清洗质量、前后端分工、性能表现、端到端可运行性、代码可复现性 |
| 案例与洞察 | 4% | 能否通过系统发现数据中的 non-trivial pattern，并在文档中形成有说服力的 case study |

### 4. 展示与答辩（10%）

| 评分项 | 分值 | 说明 |
|--------|:----:|------|
| Demo 视频 | 2% | 3~5 分钟，展示系统核心功能与典型使用流程 |
| 现场讲述 | 3% | 设计动机、核心创新点、技术难点的表达清晰度 |
| 现场操作演示 | 2% | 系统在现场能否稳定运行并响应操作 |
| 答辩问答 | 3% | 对设计决策的辩护能力、组员对各自负责部分的掌握程度 |
