# 2026 埃博拉疫情时空可视化分析系统 — 设计文档

> 设计日期：2026-05-31 | 团队：3 人 | 部署目标：GitHub Pages

## 一、项目概述

设计开发一个多视图联动的疫情时空可视化分析系统，面向 2026 年刚果（金）与乌干达跨境埃博拉疫情（本迪布焦型）。系统以**多视图实时联动（Brushing & Linking）**为核心亮点，覆盖时空分布、时序趋势、多因素关联、政策效果四个分析维度。

## 二、总体架构

```
┌─────────────────────────────────────────────────────────┐
│                    展示层 (View Layer)                    │
│  ┌──────────┐ ┌──────────┐ ┌────────┐ ┌──────────┐     │
│  │ 时空热力图 │ │ 时序趋势图 │ │平行坐标图│ │政策时间轴 │     │
│  └────┬─────┘ └────┬─────┘ └───┬────┘ └────┬─────┘     │
│       └────────────┴───────────┴────────────┘            │
│                        │                                 │
│               ┌────────▼────────┐                        │
│               │   事件分发层     │  ← 用户交互→事件统一处理 │
│               └────────┬────────┘                        │
├────────────────────────┼─────────────────────────────────┤
│                    状态层 (State Layer)                    │
│               ┌────────▼────────┐                        │
│               │  全局 Store      │                        │
│               └────────┬────────┘                        │
├────────────────────────┼─────────────────────────────────┤
│                    数据层 (Data Layer)                    │
│  ┌─────────────────────▼──────────────────────────┐      │
│  │  预处理后的静态 JSON/CSV 文件                    │      │
│  └────────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

**设计原则**：
- 视图层只管渲染，不持有筛选状态
- 所有交互走 `dispatch(action) → Store 更新 → 订阅者重绘`
- 数据预处理在 Python 中完成，前端加载静态文件

## 三、全局状态设计

```javascript
store = {
  // 时间维度
  timeRange: [startDate, endDate],      // 当前选中时间段
  animatingDate: null,                   // 热力图播放的当前日期
  isPlaying: false,                     // 播放/暂停状态

  // 空间维度
  selectedRegions: [],                  // 持久筛选（点击/框选 → 全局联动）
  highlightedRegions: [],               // 瞬时高亮（悬浮 → 仅视觉反馈）

  // 多维分析维度
  parallelAxesFilter: {
    populationDensity: [min, max],
    caseCount: [min, max],
    mortalityRate: [min, max],
    healthcareAccess: [min, max],
  },

  // 政策维度
  selectedPolicyIds: [],                // 选中的政策事件节点
}
```

### 联动规则矩阵

| 用户操作 | → 地图 | → 时序图 | → 平行坐标 | → 政策轴 | → 详情面板 |
|----------|:-----:|:------:|:--------:|:------:|:--------:|
| 地图框选/点击区域 | — | 仅显示该区域曲线 | 高亮该区域样本 | 显示该区域政策 | 更新统计 |
| 时序图 dataZoom 刷选 | 仅该时段病例 | — | 过滤到该时段 | 高亮时段内政策 | 更新统计 |
| 平行坐标刷选维度 | 仅符合条件区域 | 仅符合条件区域曲线 | — | 显示对应政策 | 更新统计 |
| 政策轴点击节点 | 受影响区域变化 | 标注前后对比区间 | — | — | 列出政策详情 |
| 播放按钮 | 逐日更新热力 | 推进指示线 | 跟随过滤 | 跟随 | 更新统计 |
| 悬浮省份 | 视觉高亮 | 对应曲线加粗 | — | — | — |

> `selectedRegions` 触发全局数据过滤；`highlightedRegions` 仅触发视觉增强（加粗/高亮），不影响筛选结果。

## 四、技术选型

| 层 | 技术 | 理由 |
|----|------|------|
| 图表 | ECharts 5 | 内置 map、parallel、brush、dataZoom，原生联动 |
| 地图底图 | Leaflet + ECharts 叠加 | 真实地理参考 + 区域热力 |
| 状态管理 | 手写 createStore（~50行） | 零依赖、答辩可解释 |
| 样式/布局 | CSS Grid | 五视图分区，无框架依赖 |
| 部署 | GitHub Pages | 静态文件直出，助教打开即用 |
| 数据预处理 | Python (Pandas, NumPy) | 离线运行，产出 data/*.json |

## 七、数据策略

采用**三层构造策略**：

| 层级 | 数据类型 | 来源 | 性质 |
|:----:|----------|------|:----:|
| 第一层 | 病例数据（确诊/死亡/康复） | WHO AFRO、ReliefWeb、ACAPS | 完全真实 |
| 第二层 | 医疗资源（医院数、床位、医护） | WHO GHO 全国数据 → 按人口密度/经济水平推算到省级 | 推算填充，注明公式 |
| 第三层 | 地理人口（GeoJSON、人口密度） | HDX、World Bank | 完全真实 |
| 第四层 | 政策事件（封锁/疫苗/援助节点） | ACAPS、ReliefWeb 新闻逐一提取 | 完全真实，标注来源链接 |

预计数据规模：病例记录 3000~5000 条，省级地理单元 ~100 个，政策事件 50~80 条。

## 十、设计决策记录

以下为团队独立做出的关键设计决策（答辩时可清晰解释）：

1. **集中式状态管理**：选择手写 Store 而非引入状态库，保证联动一致性同时保持代码可解释
2. **selectedRegions vs highlightedRegions 分离**：持久筛选与瞬时高亮解耦，避免悬浮操作触发不必要的全局重绘
3. **dataZoom 作为唯一时间过滤器**：复用 ECharts 内置能力，避免标题栏滑块与时序图功能重叠
4. **纯前端静态部署**：预处理数据为 JSON，无需后端，助教 15 分钟内可完成环境搭建
5. **三层数据构造策略**：核心真实、补充可追溯、学术上清晰可接受
