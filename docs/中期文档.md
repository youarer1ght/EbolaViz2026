# 2026 年埃博拉疫情时空可视化分析系统 — 中期文档

> 数据可视化导论课程大作业 | 3 人团队 | 2026 年 6 月 2 日

## 第一章 绪论

### 1.1 研究背景

2026 年 3 月，刚果（金）东部伊图里省（Ituri）出现不明原因高死亡率疾病聚集性病例。2026 年 5 月 13 日，刚果（金）国家生物医学研究所（INRB）在金沙萨实验室确诊 8 例本迪布焦型埃博拉病毒（Bundibugyo ebolavirus, BDBV）阳性。5 月 15 日，刚果（金）与乌干达先后官方宣布疫情爆发。5 月 16 日，世界卫生组织（WHO）宣布此次疫情构成"国际关注的突发公共卫生事件"（PHEIC）。

本次疫情具有以下显著特点：

- **罕见病毒株**：本迪布焦型埃博拉病毒此前仅记录过两次爆发（2007 年乌干达、2012 年刚果金），**尚无获批疫苗或特效治疗药物**；
- **跨境传播**：疫情在刚果（金）东部与乌干达边境地区同时蔓延，已波及伊图里（Ituri）、北基伍（Nord-Kivu）、南基伍（Sud-Kivu）三省 13 个卫生区，乌干达首都坎帕拉亦报告输入病例；
- **检测延迟**：病毒约在社区中传播 4 周后才被实验室确诊，导致较广泛的地理扩散；
- **安全局势复杂**：东部疫区存在持续武装冲突，已发生多起针对埃博拉治疗中心的袭击事件，超过 10 万人流离失所。

截至 2026 年 5 月 29 日，刚果（金）报告 977 例以上疑似病例、228 例以上疑似死亡；实验室确诊 134 例（刚果金 125 例，乌干达 9 例），确诊死亡 18 例（WHO DON 605；WHO AFRO SitRep 02）。

传统的表格统计与静态数据报表难以直观展示疫情的**时空扩散规律、跨境传播路径、区域风险差异**以及**防控政策干预效果**。借助数据可视化技术，能够将多源公共卫生数据整合为交互式分析视图，直观挖掘疫情传播特征，为公共卫生防控分析提供可视化支撑。

### 1.2 研究意义

**理论意义**：基于多源真实公共卫生数据（WHO AFRO 疫情周报、World Bank 人口统计、ReliefWeb 人道主义资讯），构建疫情可视化分析体系，丰富时空数据可视化在突发传染病领域的应用实践，特别是针对无疫苗可用的罕见病毒株疫情的可视化分析范式。

**实际意义**：直观展示埃博拉疫情时空扩散路径、高风险卫生区域、医疗资源分布短板与防控政策介入效果，帮助公共卫生研究者与决策者理解跨境传染病传播规律，具备现实参考价值。

### 1.3 国内外研究现状

传统疫情分析多以统计图表、单一时序曲线为主，存在以下不足：

- **视图单一**：仅展示时间维度（如每日新增曲线），缺乏空间维度与多因素维度的联动；
- **缺乏时空联动**：无法在同一分析环境中同时观察"在哪里发生"与"何时发生"；
- **多维度关联缺失**：难以将病例数据与人口密度、医疗资源、政策事件等多源数据进行关联分析；
- **现有埃博拉可视化研究多针对历史疫情**（如 2014-2016 西非埃博拉、2018-2020 刚果金埃博拉），缺少对 2026 年最新本迪布焦型跨境疫情的**多视图联动分析**。

### 1.4 研究内容与技术路线

**研究内容**：数据预处理、疫情时空分布分析、影响因素关联分析、防控政策效果可视化、多视图联动交互设计。

**技术路线**：数据获取（WHO/ReliefWeb/World Bank/HDX）→ 数据清洗与融合 → 数据分析建模 → 多视图可视化设计 → 联动交互实现 → 结果分析与总结。

### 1.5 论文组织结构

本文共七章。第一章介绍研究背景与意义；第二章概述相关技术与理论基础；第三章介绍数据集与预处理方案；第四章阐述可视化需求分析与总体设计；第五章为可视化详细设计与实现；第六章为可视化结果分析与讨论；第七章总结与展望。

## 第二章 相关技术与理论基础

### 2.1 数据可视化基础理论

本系统涉及以下可视化核心理论：

- **时空数据可视化**：将时间序列数据与地理空间数据结合展示，通过热力图、时间轴播放等方式呈现疫情的时空传播规律；
- **多视图协同（Multiple Coordinated Views）**：多个视图共享同一数据源，各自展示数据的不同维度，用户在一个视图中的操作同步反映到其他视图中；
- **刷选联动（Brushing & Linking）**：用户在任一视图中刷选数据子集，其他视图同步高亮或过滤对应的数据项，是多视图协同的核心交互机制；
- **细节按需展示（Details-on-Demand）**：用户可通过悬浮、点击等操作获取具体数据的详细属性信息。

### 2.2 开发技术与工具

| 层 | 技术选型 | 说明 |
|----|---------|------|
| 图表渲染 | ECharts 5 | 内置 map、parallel、brush、dataZoom 等组件，原生支持多视图联动 |
| 地图底图 | ECharts registerMap（GeoJSON） | 直接绘制区域热力图，简化实现，避免多框架坐标系对齐问题 |
| 状态管理 | 手写 createStore（观察者模式，~50 行） | 零外部依赖，答辩可完整解释每一行的设计逻辑 |
| 页面布局 | CSS Grid | 五视图分区布局，无 UI 框架依赖 |
| 数据处理 | Python（Pandas, NumPy） | 离线预处理，产出静态 JSON 文件供前端直接加载 |
| 部署方案 | GitHub Pages | 静态文件托管，助教打开浏览器即可使用，15 分钟内可完成环境搭建 |

### 2.3 疫情可视化设计原则

- **真实性原则**：数据来源于 WHO AFRO、World Bank、ReliefWeb 等权威平台，所有数据均可溯源；
- **直观性原则**：采用热力图渐变、多色曲线、分类色标等视觉通道，让用户快速感知数据模式；
- **多维度联动原则**：所有视图通过集中式状态管理实现实时联动，选时间、选区域、选维度，所有图表同步更新；
- **交互性原则**：支持点击选择、框选过滤、时间播放、悬浮高亮等多种交互方式；
- **简洁可读性原则**：采用浅灰背景减少视觉疲劳，视图分区清晰，配色遵循 Tableau 10 等经典方案。

## 第三章 数据集与数据预处理

### 3.1 数据集介绍

本项目采用"真实数据为主、推算填充为辅"的三层数据构造策略，共涉及五类核心数据：

#### B.1 核心疫情病例数据（主数据）

- **数据集名称**：2026 年刚果（金）、乌干达本迪布焦型埃博拉疫情官方数据集
- **来源平台**：[WHO AFRO 每周外部情况报告](https://www.afro.who.int/countries/democratic-republic-of-congo/publication/ebola-bundibugyo-virus-disease-outbreak-0)（SitRep 01 发布于 2026-05-18；SitRep 02 发布于 2026-05-24），数据经 [kraemer-lab/Ebola_DRC_2026](https://github.com/kraemer-lab/Ebola_DRC_2026) 结构化整理
- **数据内容**：每日疫区确诊、死亡、疑似病例数，按卫生区（health zone）统计
- **许可**：WHO 开放获取（需署名）

#### B.2 疫情官方通报与事件数据

- **来源平台**：[WHO 疾病爆发新闻（DON 605）](https://www.who.int/emergencies/disease-outbreak-news/item/2026-DON605)、[ReliefWeb](https://reliefweb.int/)
- **数据内容**：疫情爆发节点、跨境传播事件、WHO 官方响应措施、边境关闭、疫苗接种、国际援助等关键事件
- **处理方式**：从新闻通报中逐一提取，结构化标注时间、类型、地区、来源链接

#### B.3 地理、人口与行政区划数据

- **来源平台**：[World Bank 公开数据](https://data.worldbank.org/)、[HDX 人道主义数据交换平台](https://data.humdata.org/)
- **数据内容**：刚果（金）、乌干达疫区行政区划边界、经纬度、人口密度、城乡分布、边境口岸地理坐标
- **许可**：CC BY 4.0

#### B.4 医疗资源统计数据

- **来源平台**：[WHO 全球卫生人力统计数据库](https://www.who.int/data/gho)、[GRID3 卫生设施数据](https://data.grid3.org/)
- **数据内容**：疫区每万人医生数、每万人床位数、卫生站点数量
- **处理方式**：以国家级宏观数据为基础，结合各地人口密度、经济水平推算到卫生区级别；推算公式与假设在系统文档中明确注明

#### B.5 防控政策与人道主义资讯数据

- **来源平台**：[ReliefWeb 全球人道主义资讯平台](https://reliefweb.int/)
- **数据内容**：各国隔离管控措施、边境筛查政策、国际援助金额、人道主义行动节点

### 3.2 数据存在问题

1. **多源数据格式不统一**：WHO SitRep 为 PDF 格式，需提取为表格；World Bank 数据需按行政区划映射；
2. **行政区划名称不匹配**：不同来源的同一地区可能有不同拼写（如 "Mongbwalu" vs "Mongbalu"、"Rwampara" vs "Rwmapara"），通过 kraemer-lab 的 aliases.csv 进行名称规范化；
3. **时间粒度不统一**：WHO SitRep 为每周发布，非每日更新；政策事件为具体日期；
4. **部分偏远地区数据缺失**：部分卫生区的医疗资源数据需通过推算获得。

### 3.3 数据预处理与融合

数据预处理流程如下：

1. **缺失值处理**：对于医疗资源缺失的偏远卫生区，采用"国家均值 × 区域人口密度比 × 调整系数"的推算公式填充；
2. **异常值剔除**：对超出合理范围的数值进行人工核查；
3. **时间粒度统一**：将不同时间粒度的数据统一到日期级别；
4. **行政区划字段匹配**：通过 `aliases.csv` 映射表将所有地区名称统一为规范名称；
5. **多表数据融合**：按 `region + date` 进行外连接，生成标准化分析数据集；
6. **输出格式**：导出为 `data/*.json` 文件，前端通过 `fetch()` 直接加载。

预处理的执行脚本为 `scripts/build_real_data.py`。

## 第四章 可视化总体设计与视图规划

### 4.1 用户需求分析

面向课程分析需求与公共卫生分析场景，本系统需满足以下分析需求：

- **时空规律观察**：疫情在何时、何地爆发？如何随时间扩散？
- **风险因素挖掘**：哪些地区的死亡率偏高？与人口密度、医疗资源是否相关？
- **政策效果对比**：边境封锁、疫苗接种、国际援助等干预措施是否改变了疫情走势？
- **交互探索分析**：用户能否自由筛选时间段、区域、维度，自主探索数据中的模式？

### 4.2 功能性需求

| 需求类型 | 具体功能 |
|----------|---------|
| 时空分布展示 | 疫情地理热力图，展示各卫生区病例分布与热点区域 |
| 时序趋势分析 | 多地区病例时序曲线，支持时间范围缩放与刷选 |
| 多因素关联分析 | 平行坐标图，关联人口密度、医疗资源、病例数、死亡率等维度 |
| 政策对比分析 | 政策事件时间轴，对比政策前后疫情走势差异 |
| 多视图联动交互 | 所有视图通过 Brush & Link 机制实时联动 |
| 细节查询展示 | 统计面板展示当前筛选条件下的关键指标与区域排名 |

### 4.3 非功能性需求

- 界面简洁直观，视图分区清晰
- 交互流畅，状态更新延迟 < 100ms
- 数据准确，所有指标可从原始数据追溯
- 视图层次清晰，支持响应式布局
- 可拓展性强，新增视图只需订阅 Store 即可加入联动体系

### 4.4 总体设计架构

本系统采用四层架构：

```
┌──────────────────────────────────────┐
│  展示层 (View Layer)                  │
│  ① 时空热力图  ② 时序趋势图           │
│  ③ 平行坐标图  ④ 政策时间轴  ⑤ 详情面板│
├──────────────────────────────────────┤
│  事件分发层: 用户交互 → dispatch(action) │
├──────────────────────────────────────┤
│  状态层 (State Layer): 全局 Store      │
│  timeRange / selectedRegions /        │
│  parallelAxesFilter / selectedPolicyIds│
├──────────────────────────────────────┤
│  数据层 (Data Layer): 静态 JSON 文件     │
└──────────────────────────────────────┘
```

**核心设计原则**：
- 视图层只管渲染，不持有筛选状态；
- 所有用户交互统一走 `dispatch(action) → Store 更新 → 所有视图订阅者重绘`；
- 数据预处理在 Python 中离线完成，前端直接加载静态 JSON 文件。

### 4.5 可视化视图总体布局设计

采用 CSS Grid 实现五视图分区布局：

```
┌──────────────────────────────────────────────────────────┐
│  标题栏：系统标题 + 播放/暂停/重置 + 当前播放日期          │
├──────────────────────┬───────────────────────────────────┤
│   ① 时空热力图        │   ② 时序趋势图                     │
│   (55% × 50%)         │   (45% × 50%)                     │
├──────────────────────┴───────────────────────────────────┤
│   ③ 平行坐标图（多因素关联）    (100% × 30%)              │
├──────────────────────┬───────────────────────────────────┤
│   ④ 政策时间轴        │   ⑤ 详情面板                      │
│   (40% × 20%)         │   (60% × 20%)                     │
└──────────────────────┴───────────────────────────────────┘
```

CSS Grid: `grid-template-columns: 55fr 45fr; grid-template-rows: auto 50fr 30fr 20fr;`

## 第五章 可视化详细设计——多视图联动方案（核心）

### 5.1 全局状态设计

系统采用集中式状态管理（手写观察者模式 Store，约 50 行），所有筛选条件统一存储：

```javascript
store = {
  timeRange:          [startDate, endDate],   // 时间范围
  animatingDate:      null,                    // 热力图播放当前日期
  isPlaying:          false,                   // 播放/暂停
  selectedRegions:    [],                      // 持久筛选（点击/框选 → 全部联动）
  highlightedRegions: [],                      // 瞬时高亮（悬浮 → 仅视觉反馈）
  parallelAxesFilter: { ... },                 // 平行坐标各维度刷选范围
  selectedPolicyIds:  [],                      // 选中的政策事件
}
```

**设计决策**：`selectedRegions` 与 `highlightedRegions` 分离——前者触发全局数据过滤（持久），后者仅触发视觉增强（瞬时），避免悬浮操作引发不必要的全局重绘。

### 5.2 五视图联动规则

| 用户操作 | → 热力图 | → 时序图 | → 平行坐标 | → 政策轴 | → 详情面板 |
|----------|:------:|:------:|:--------:|:------:|:--------:|
| 地图点击/框选区域 | — | 仅显示该区域曲线 | 高亮该区域样本 | 显示该区域政策 | 更新统计 |
| 时序图 dataZoom 刷选 | 仅该时段 | — | 过滤到该时段 | 高亮时段内政策 | 更新统计 |
| 平行坐标维度刷选 | 仅符合条件区域 | 仅符合条件区域 | — | 显示对应政策 | 更新统计 |
| 政策轴点击节点 | 受影响区域 | 标注前后对比 | — | — | 列出政策详情 |
| 播放按钮 | 逐日更新 | 推进指示线 | 跟随过滤 | 跟随 | 更新统计 |
| 悬浮省份 | 视觉高亮 | 曲线加粗 | — | — | — |

### 5.3 配色方案

| 用途 | 配色 | 说明 |
|------|------|------|
| 热力图 | 黄→橙→红 5 级渐变 | 风险感知直觉；备选蓝→橙（色盲友好） |
| 时序曲线 | Tableau 10 调色板 | 多地区高区分度 |
| 平行坐标 | 按死亡率渐变着色 | 突出高风险数据路径 |
| 政策轴 | 红(封锁)/绿(疫苗)/蓝(援助)/橙(监测)/紫(响应) | 快速区分类型 |
| 背景 | `#f5f7fa` 浅灰 | 减少视觉疲劳 |

## 中期小结

本项目目前已完成了系统的总体架构设计与核心技术选型。在数据层面，建立了以 WHO AFRO 周报数据为核心、World Bank 人口数据与 ReliefWeb 政策事件数据为补充的真实数据体系。在技术层面，实现了基于集中式 Store 的五视图联动框架，所有视图共享统一状态，用户在任何视图中的交互操作均通过 Store 自动同步到其他视图。
