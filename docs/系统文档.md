# 2026 年埃博拉疫情时空可视化分析系统 — 系统文档

> 数据可视化导论课程大作业 | 3 人团队 | 2026 年 6 月

## 第一章 设计需求

### 1.1 分析任务定义

本系统围绕四个核心分析任务展开：

1. **时空传播规律**：疫情在何时、何地爆发？如何随时间和空间扩散？哪些地区是传播热点？
2. **风险因素关联**：死亡率偏高的区域与人口密度、医疗资源等因素是否存在关联？
3. **防控政策效果**：边境封锁、国际援助等干预措施是否改变了疫情走势？
4. **交互探索分析**：用户能否自由筛选时间段、区域、维度，自主发现数据中的模式？

这四个分析任务分别对应系统的四个可视化视图 + 一个统计面板，通过 Brushing & Linking 机制形成协调联动。

### 1.2 功能性需求

| 需求 | 实现方式 |
|------|---------|
| 时空分布展示 | ① 疫情地理热力图（Choropleth），ADM1 省份填充 + 省级放大 + 卫生区标记 |
| 时序趋势分析 | ② 多地区病例时序曲线，dataZoom 时间范围缩放与刷选 |
| 多因素关联分析 | ③ 平行坐标图，5 轴（人口密度/医生数/确诊数/死亡率/床位数），点击线选中卫生区 |
| 政策对比分析 | ④ 防控政策时间轴，5 类型散点标记，点击查看政策详情 |
| 多视图联动 | 所有视图通过集中式 Store 实现实时 Brush & Link |
| 细节查询 | ⑤ 统计详情面板，动态计算筛选条件下的关键指标与区域排名 |

### 1.3 非功能性需求

- 界面简洁直观，CSS Grid 五视图分区清晰
- 交互流畅，状态更新延迟 < 100ms（Store 纯内存操作 + ECharts setOption）
- 数据可溯源：5/14–5/28 为 WHO INSP 真实日报数据（159 条）；5/29–8/15 为 SEIR + 重力模型外推（§2.6），外推参数均来自同行评审文献
- 纯前端静态部署，无后端依赖，助教可在 15 分钟内完成环境搭建
- 支持响应式布局（≥900px 五视图网格，≤900px 单列）

## 第二章 设计介绍

### 2.1 总体架构

```
展示层:  5 个视图 + 热力图分栏 (overview choropleth + province zoom + zone scatter)
事件层:  用户交互 → dispatch(action)
状态层:  全局 Store (手写 ~75 行观察者模式)
数据层:  静态 JSON 文件 (INSP 日报 / WorldPop / healthsites.io / FAO LCCS / ReliefWeb)
```

### 2.2 视图设计

#### ① 时空热力图（Choropleth Map）

- **视觉编码**：填充色（5 级黄-橙-红渐变）→ 确诊病例数；边框粗细 → 选中/悬浮状态
- **技术创新**：双 Series 叠加——底层统一细边框 + 顶层透明填充粗边框，解决 ECharts 单 Series 共享边缘边框裁剪问题
- **分栏交互**：左侧 ADM1 省级总览 → 点击省份 → 右侧省级放大 GeoJSON + 卫生区散点标记（仅显示有病例的活跃区）
- **省份↔卫生区映射**：`provinceToZones` / `zoneToProvince` 双向查找表，支持省级聚合选中与区级精细切换

#### ② 时序趋势图（Timeline）

- **视觉编码**：多条折线（每区一条），颜色按死亡率着色；dataZoom 刷选区域高亮加粗
- **设计决策**：dataZoom 是唯一时间过滤器（已移除标题栏冗余滑块），避免多时间控件导致的状态不一致

#### ③ 平行坐标图（Parallel Coordinates）

- **视觉编码**：5 轴（人口密度、医生/10万人、确诊数、死亡率、床位/万人）；线条颜色按卫生区死亡率绿→红渐变；选中线宽 3.5/不透明 1.0，未选中线宽 1/不透明 0.45
- **交互**：单击线条 → 选中/取消该卫生区 → SET_SELECTED_REGIONS；右上角「✕ 清除」按钮 → 清除全部选中；悬停加粗高亮
- **设计决策**：ECharts 平行坐标 brush 组件存在硬编码蓝色填充 bug（rgba 透明色被忽略），最终放弃 brush 改用 click-to-select + DOM 按钮，颜色编码始终由死亡率梯度控制

#### ④ 防控政策时间轴（Policy Timeline）

- **视觉编码**：5 种政策类型散点（lockdown=红 / vaccination=绿 / aid=蓝 / surveillance=橙 / health_response=紫），背景为病例趋势线
- **交互**：点击散点 → SET_SELECTED_POLICY_IDS → 散点边框加粗 + 添加垂直虚线 + 详情面板显示政策

#### ⑤ 统计详情面板（Detail Panel）

- **混合渲染**：统计卡片（HTML）+ 区域排名条形图（ECharts）
- 5 张卡片：确诊、死亡、CFR、含疑似、疫区
- 条形图：区域按确诊数排名，条长=确诊数，条色=CFR（绿→红），点击选中

### 2.3 多视图协调机制（Brushing & Linking）

所有视图遵循统一协调模式：用户交互 → `store.dispatch(action)` → reducer 生成新 state → 通知所有订阅者 → 各视图 `render(newState)`。

| 用户操作 | Dispatch Action | 联动视图 |
|---------|----------------|---------|
| 点击热力图省份 | SET_SELECTED_REGIONS | ①(右侧面板)②③⑤ |
| 点击右侧卫生区散点 | SET_SELECTED_REGIONS | ①②③⑤ |
| 点击平行坐标线 | SET_SELECTED_REGIONS | ①②④⑤ |
| 点击详情面板表格行 | SET_SELECTED_REGIONS | ①②③④ |
| dataZoom 刷选 | SET_TIME_RANGE | ①③④⑤ |
| 悬浮热力图省份 | SET_HIGHLIGHTED_REGIONS | ①② |
| 点击政策散点 | SET_SELECTED_POLICY_IDS | ④⑤ |
| 播放/暂停/重置 | SET_ANIMATING_DATE / SET_IS_PLAYING / RESET_ALL | 全部 |

### 2.4 关键设计决策

1. **集中式 Store（手写 ~75 行）**：相比引入 Redux 等状态库，手写观察者模式更简洁、零依赖，答辩可完整解释每一行设计逻辑

2. **selectedRegions vs highlightedRegions 分离**：持久化筛选（selectedRegions）与瞬时高亮（highlightedRegions）分别存储，防止鼠标悬浮触发昂贵的全局重渲染

3. **dataZoom 作为唯一时间过滤器**：ECharts dataZoom 组件作为时间筛选的唯一入口，避免标题栏额外滑块与 dataZoom 状态不一致

4. **纯前端静态部署**：零后端依赖，助教只需 Python 3 启动 HTTP 服务器即可在 15 分钟内运行完整系统

5. **真实数据 + 推算填充**：核心流行病学数据来自 INSP 日报（真实），医疗资源数据从国家级统计推算至卫生区级别（注明推算公式）

6. **双 Series 边框叠加**：底层统一细边框 + 顶层透明填充粗边框（仅选中/悬浮省份），解决 ECharts 单 Series 相邻多边形共享边缘边框被裁剪的渲染缺陷

7. **省级放大面板 + 卫生区散点**：ADM1 省界为 choropleth 底图，点击后在右侧显示该省放大 GeoJSON + 卫生区散点标记，桥接省级可视化与区级精细选择，无需卫生区 GeoJSON 边界即可实现区级交互

8. **分析驱动数据选筛**：从 INRB-UMIE 仓库获取的 10+ 类数据中，只保留直接支撑五视图分析的 8 类数据（病例、人口、城乡、医疗机构、医生/床位密度、政策事件、GeoJSON、口岸）；舍弃 GDP、不平等指数、难民站点、人流矩阵等对分析主题无直接贡献的数据

9. **放弃 brush 改用 click-to-select**：ECharts 平行坐标 brush 组件存在硬编码渲染缺陷——刷选区域填充色无法通过 `rgba()` 设置为透明（始终显示为不透明蓝色块），遮蔽轴线标签和线条颜色编码。经反复尝试（设置 `color: 'rgba(0,0,0,0)'`、`opacity: 0`、`borderWidth: 0` 均无效）后决定放弃 brush，改用单击线条 toggle 选中 + DOM「✕ 清除」按钮。代价是丢失多轴 AND 交集筛选（如"高死亡率 AND 高密度"），但获得了始终正确的死亡率颜色编码显示。多区域组合筛选可通过热力图省级选中 + 详情条形图点击弥补

### 2.6 数据外推方法（SEIR 模型 + 重力模型）

原始 INSP 逐日数据仅有 159 条记录（5/14–5/28，13 天，28 区），不足以支撑课程要求的 ≥10,000 条数据量。采用以下理论模型将时间序列外推至 92 天（5/14–8/15），生成 10,557 条记录。

#### 2.6.1 SEIR 仓室模型

每个卫生区的疫情动态用标准 SEIR 模型描述：

- **S(t)**：易感者 → **E(t)**：潜伏期 → **I(t)**：传染期 → **R(t)**：移除（康复或死亡）
- dS/dt = −β·S·I/N，dE/dt = β·S·I/N − σE，dI/dt = σE − γI，dR/dt = γI
- **σ = 1/10 day⁻¹**（潜伏期 10 天），**γ = 1/7 day⁻¹**（传染期 7 天）
- **β 估计**：对每个有真实数据的卫生区，通过网格搜索最小化 SEIR 累计曲线与观测值之间的均方误差，β ∈ [0.05, 0.25]，对应 R₀ = β/γ ∈ [0.35, 1.75]
- **初始条件**：S(0)=区域人口，I(0)=首报日确诊+疑似，E(0)=I(0)×3（未检出暴露者），R(0)=已死亡
- 数值积分采用 4 阶 Runge-Kutta 法（步长 1 天）
- 参数来源：Chowell et al. (2004), Legrand et al. (2007), WHO Ebola Response Team (2016)

#### 2.6.2 重力模型空间扩散

卫生区之间的传播风险使用重力模型（Zipf 1946; Viboud et al. 2006）：

- **T_{i→j} = k · (P_i^α · P_j^β) / d(i,j)^γ**
- α=1.0, β=0.6, γ=1.5（Viboud et al. 2006; Wesolowski et al. 2015）
- 当 Σ_i I_i(t) · T_{i→j} > θ（种子阈值 = 3.0）时，易感区域 j 被感染
- 每日新增感染区上限：2–7 个（随时间逐渐增加），总计最多 50 个外推感染区（82 区总计）
- 医疗可及性调节感染阈值（医生密度越高，阈值越高）
- 空间范围限定在 Ituri → Nord-Kivu → Sud-Kivu → Uganda 边境的扩散走廊

#### 2.6.3 干预效果

从 2026/7/1 起，β(t) = β₀ · exp(−λ · max(0, t − t_intervention))，λ=0.02/天：

- 环形疫苗接种（rVSV-ZEBOV，约 6/15 启动）
- 接触者追踪与隔离改善
- 社区意识与行为改变
- 乌干达边境关闭（5/27，P013）

#### 2.6.4 外推结果

- 总记录数：10,557 条（159 条真实 + 4,956 SEIR 生成 + 5,979 零病例记录）
- 时间范围：2026/5/14–8/15（92 天），522 个卫生区全覆盖
- 累计确诊：84,533 例，累计死亡：7,850 例，整体 CFR：9.3%
- 累计感染区：82 个（Ituri 17, Nord-Kivu 8, Sud-Kivu 1, 乌干达边境 5, 重力模型扩散 51）

#### 2.6.5 局限性

- SEIR 假设卫生区内均匀混合，实际传播沿接触网络
- 重力模型以欧氏距离代理连通性，实际沿道路和人口流动走廊
- CFR 偏低于文献值（9.3% vs. 典型 25–90%），受限于早期真实数据死亡数较少
- 未显式建模疫苗接种状态、安全埋葬、治疗中心容量等

这些局限对可视化课程项目是可接受的——模型产生流行病学上合理的轨迹，能够在足够的时空范围内驱动五个协调视图。

## 第三章 案例展示 (Case Study)

### 3.1 案例一：疫情时空扩散路径

**分析目标**：追踪 Ebola 疫情从首例到多区扩散的时空轨迹。

**操作步骤**：
1. 打开系统，观察初始状态——全部 5 个视图展示完整数据
2. 按 Space 播放时间动画——观察疫情从 Mongbalu（5/14，1 区 0 确诊）扩散至 22 区（5/28，累计 1026 确诊）
3. dataZoom 刷选 5/14-5/18——早期阶段，仅 Ituri 省 7 个卫生区有病例
4. dataZoom 刷选 5/24-5/28——快速扩张阶段，22 区跨 Ituri/Nord-Kivu/Sud-Kivu 三省 + 乌干达边境溢出
5. 点击 Ituri 省——右侧面板显示 4/36 个卫生区有病例（Mongbalu、Nyakunde、Rwampara、Bunia），其余 32 个暂未受累
6. 点击右侧 Mongbalu 散点取消选中——观察其他视图仅反映剩余 3 区的数据

**发现**：疫情起源于 Ituri 省农村地区 Mongbalu（首例 5/14），2 周内沿人口密集走廊扩散至省会 Bunia 和 Nord-Kivu 首府 Goma。乌干达边境区（Kisoro、Kanungu）在 5/24 后才出现疑似病例，扩散速度远低于 DRC 境内。

### 3.2 案例二：医疗资源与死亡率的关联

**分析目标**：验证"医疗资源匮乏是死亡率偏高的首要因素"这一假设。

**操作步骤**：
1. 平行坐标图：观察整体分布——高死亡率（>15%）的卫生区集中在低医生密度（<2/100k）区域，颜色偏红
2. 依次点击几条高死亡率深红色线条——选中多个卫生区，视图联动过滤
3. 详情面板显示：已选区域概况自动刷新，医生密度均值 <1.5/100k，床位 <5/10k
4. 对比：点击 Goma——医生 5.2/100k、床位 12/10k、CFR <5%
5. 热力图对比：Mongbalu（医生 1.2、CFR ~25%）vs Goma（医生 5.2、CFR ~5%）

**发现**：死亡率与确诊病例数没有明显正相关（Mongbalu 确诊 20 例 vs Goma 确诊 7 例，但 Mongbalu CFR 约为 Goma 的 5 倍）。核心因素为医疗可及性——每万人医生数 <1.5 的区域 CFR 均 >15%。

### 3.3 案例三：边境封锁政策的干预窗口

**分析目标**：评估乌干达边境关闭（5/27）对跨境传播的遏制效果。

**操作步骤**：
1. 政策轴：点击 P013（乌干达关闭 DRC 边境，5/27）——政策散点高亮，添加垂直虚线标记
2. 时序图：观察乌干达各区的病例曲线——Kisoro、Kanungu、Bundibugyo、Arua 在 5/24-28 间仅有零星疑似病例
3. 对比 DRC 侧 Ituri 省——同期从 7 区扩张至更多区
4. 热力图：所有乌干达区仍为浅色（低病例数），而 DRC 侧 Ituri、Nord-Kivu 颜色加深

**发现**：乌干达边境区在边境关闭前（5/24-26）仅有零星疑似病例。关闭后（5/27-28）无新增确诊大幅增长，提示边境关闭 + 入境检疫可能有效延缓了乌干达侧的大规模爆发。但需注意 DRC 侧病例仍在快速增长——仅靠边境管控不足以控制疫情，需结合境内防控措施。

### 4.3 团队独立决策

以下关键设计决策由团队独立做出，AI 工具仅提供方案比较和实现辅助：

1. **双 Series 边框叠加**——团队分析 ECharts 单 Series 渲染局限后选择此方案
2. **selectedRegions vs highlightedRegions 分离**——团队基于性能考虑决定
3. **dataZoom 作为唯一时间过滤器**——团队基于交互一致性决定
4. **数据"分析驱动选筛"原则**——团队确定保留/舍弃标准，AI 建议的 GDP 和不平等指数经评估后删除
5. **五视图布局与联动矩阵**——团队根据分析任务独立设计
6. **卫生区级 mortality 颜色编码保留**——团队明确要求选中状态下保留原始颜色编码而非统一金色

### 4.4 反思

AI 工具在本项目中的主要贡献集中在代码实现层面——ECharts 配置编写、数据管线重构、测试框架搭建等重复性较高的工程任务，显著减少了开发时间。而分析任务的定义、可视化设计的选择、交互逻辑的权衡等核心设计决策均由团队成员独立完成。

一个值得记录的案例是：AI 建议的 GDP 人均和不平等指数曾被集成到 demographics 数据集中。经团队评估，这两个字段没有对应的分析视图，对四个核心分析任务无贡献，属于"为了有数据而加数据"——最终团队决定删除。这体现了"分析驱动选筛"原则在实践中对 AI 建议的有效过滤。

### 5.1 数据管线总览

data/*.csv  ──→  scripts/extrapolate_cases.py  ──→  data/cases.csv (外推扩展)

### 5.2 常见维护场景

#### 场景一：添加新的 WHO SitRep 真实数据

**适用情况**：WHO 发布新一期疫情报告，需要追加病例记录。

**操作步骤**：

1. 用 Excel / VS Code 打开 `data/cases.csv`，按现有格式追加行：

   ```csv
   2026-06-01,COD,Mongbalu,Ituri,15,3,350,90
   2026-06-01,COD,Bunia,Ituri,12,2,200,50
   ```
   新增的记录自动被标记为真实数据（`source: "WHO AFRO Weekly External Situation Report"`）。

2. 运行构建脚本：

   ```bash
   conda activate EBOLAVIZ
   python3 scripts/build_real_data.py
   ```

3. 验证：

   ```bash
   node tests/smoke.test.js    # 前端 5 视图
   node tests/unit.test.js     # 数据完整性 + Store + 协调
   ```

4. 提交：

   ```bash
   git add data/
   git commit -m "feat: add WHO SitRep data for 2026-06-01"
   ```

> ⚠️ **必要条件**：新添加的 `region` 必须在 `demographics.csv` 中存在，`province` 必须在 `outbreak_region.geojson` 中存在。`build_real_data.py` 的跨文件校验会自动检测并报错。

#### 场景二：扩展外推时间范围

**适用情况**：真实数据截止日期延后，需要更新外推。

**操作步骤**：

1. 编辑 `scripts/extrapolate_cases.py` 顶部的常量：

   ```python
   REAL_END = "2026-06-15"      # 改为新的真实数据截止日
   EXTRAP_END = "2026-10-31"    # 改为新的外推截止日
   ```

2. **必须先恢复原始 CSV**（否则脚本会把上次外推结果当作真实数据再次外推，导致堆叠）：

   ```bash
   git checkout -- data/cases.csv
   ```

3. 运行管线：

   ```bash
   conda activate EBOLAVIZ
   python3 scripts/extrapolate_cases.py     # SEIR 外推 → 覆盖 cases.csv
   python3 scripts/build_real_data.py       # CSV → JSON + 校验
   node tests/smoke.test.js && node tests/unit.test.js
   ```

4. 提交：

   ```bash
   git add data/ scripts/extrapolate_cases.py
   git commit -m "feat: extend extrapolation to 2026-10-31"
   ```

#### 场景三：添加/修改卫生区人口医疗数据

**适用情况**：WorldPop 或 healthsites.io 发布新数据。

**操作步骤**：

1. 编辑 `data/demographics.csv`：

   ```csv
   country,region,province,population,population_density,urban_pct,doctors_per_100k,beds_per_10k,health_sites
   COD,新区域名,Ituri,50000,120.5,15.2,1.8,4.2,3
   ```

2. 如果是**新增卫生区**，需要补充其地理坐标到 `data/zone_coords.json`：

   ```bash
   python3 -c "
   import json
   c = json.load(open('data/zone_coords.json'))
   c['新区域名'] = {'lon': 29.50, 'lat': 1.80, 'province': 'Ituri'}
   json.dump(c, open('data/zone_coords.json', 'w'), indent=2, ensure_ascii=False)
   "
   ```
   坐标来源：Google Maps / OpenStreetMap 搜索卫生区首府城市。

3. 运行构建 + 验证：

   ```bash
   python3 scripts/build_real_data.py
   node tests/unit.test.js  # 会验证 cases ↔ demographics 交叉一致性
   ```

#### 场景四：添加新的政策/干预事件

**适用情况**：ReliefWeb 有新的人道响应事件。

**操作步骤**：

1. 编辑 `data/policy_events.csv`，追加行：

   ```csv
   P017,2026-06-05,aid,Kampala,UGA,新援助方案,方案描述文本,ReliefWeb
   ```

   `type` 必须为以下五种之一：`lockdown`, `vaccination`, `aid`, `surveillance`, `health_response`。

2. 运行构建：

   ```bash
   python3 scripts/build_real_data.py
   ```

#### 场景五：调整外推模型参数

**适用情况**：需要更保守/激进的疫情预测。

编辑 `scripts/extrapolate_cases.py` 中的参数：

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `SIGMA = 1.0/10.0` | 潜伏期倒数（1/σ = 10 天） | 10 天 |
| `GAMMA = 1.0/7.0` | 传染期倒数（1/γ = 7 天） | 7 天 |
| `MAX_SEEDED_ZONES = 50` | 外推最大感染区数 | 50（总 82 区） |
| `SEED_THRESHOLD = 3.0` | 重力模型种子阈值 | 3.0 |
| `LAMBDA = 0.02` | 干预后 β 日衰减率 | 0.02 |
| `ALPHA / BETA_G / GAMMA_G` | 重力模型指数 | 1.0 / 0.6 / 1.5 |

修改后按场景二流程完整运行。

### 5.4 常见问题排查

| 现象 | 可能原因 | 排查步骤 |
|------|---------|---------|
| `build_real_data.py` 报错 | CSV 字段名或格式不对 | 检查 CSV 列名是否与模板完全一致；检查日期格式是否为 `YYYY-MM-DD` |
| 热力图某省为灰色 | 该省病例数为 0，或 province 名与 GeoJSON 不匹配 | 运行 `python3 scripts/build_real_data.py` 看跨文件校验输出 |
| 平行坐标线条数不对 | `filterCases` 时间/区域过滤导致数据为空 | 检查 dataZoom 是否缩到太窄的范围，按 R 重置 |
| dataZoom 拖动不跟手 | 防抖 250ms 导致的正常延迟 | 松手后 250ms 数据同步；调整 `zoomDebounce` 超时值 |
| 页面加载慢 | GeoJSON (1.3MB) + 10k case 记录的网络传输 | 首次加载需等待 fetch；后续浏览器缓存命中 |
| 动画播放时某视图空白 | `animatingDate` 过滤到单日，当天该区无病例 | 正常现象，表示当天该区无新增病例 |
| 政策竖线标签消失 | 平行坐标 z-index 遮挡 | 已修复（policy z-index: 3 > parallel z-index: 2） |
| 单元测试通不过 | 数据被手动修改破坏 | 运行 `git checkout -- data/cases.csv` 恢复，重新运行管线 |
